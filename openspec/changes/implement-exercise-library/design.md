## Context

The Phoenix backend is currently a Spring Boot 4.1.0 scaffold with Spring
Modulith, Spring Data JDBC, PostgreSQL and Security dependencies. The exercise
library is already documented as an MVP capability and is the first domain
module needed by workout templates.

The current OpenSpec main spec defines the behavior and contracts. This change
plans the first backend implementation only; no code should be written until
the proposal, delta spec, design and tasks are accepted.

## Goals / Non-Goals

**Goals:**

- Implement a bounded `exercise` module under
  `dev.affonsera.phoenix.exercise`.
- Persist exercises in PostgreSQL using the documented `exercises` table.
- Seed the initial catalog using stable UUIDs and idempotent logic.
- Expose read-only API endpoints for list and get-by-id.
- Add tests that verify the OpenSpec scenarios.

**Non-Goals:**

- No workout-template module.
- No workout player, history or progression engine.
- No user-owned custom exercise CRUD.
- No authentication flow design beyond using the existing Spring Security
  scaffold.
- No Redis usage.
- No infrastructure changes in `phoenix-infra`.

## Decisions

### Use a Single Spring Modulith Package Boundary

The module will live under `dev.affonsera.phoenix.exercise`. The initial module
is small enough to keep domain, repository, seed and web adapter code under one
bounded package with internal subpackages if useful.

Alternative considered: create separate `catalog`, `training` or `workout`
packages now. That would blur ownership before workout templates exist.

### Persist Enums as Strings

`movementPattern`, `bodyArea` and `equipmentType` will be represented as Java
enums and stored as uppercase strings matching OpenSpec values.

Alternative considered: PostgreSQL enum types. Text columns keep migrations
simpler during early MVP iteration while still allowing validation in Java and
database checks later if needed.

### Use UUIDs Generated by Application or Seed Data

Exercise IDs are UUIDs. Seed data will use stable UUID literals so downstream
workout templates can reference exercises without churn.

Alternative considered: database-generated numeric IDs. That makes seeded
references less portable and exposes implementation details to API clients.

### Keep API Read-Only for Initial Implementation

The first implementation exposes `GET /api/exercises` and
`GET /api/exercises/{id}` only.

Alternative considered: implement full CRUD immediately. CRUD requires product
decisions about user-owned custom exercises, permissions and audit behavior
that are not yet documented.

### Keep Seed Data in the Product Repository

Seed schema and data belong in `phoenix`, not `phoenix-infra`, because they
represent product behavior and domain defaults.

Alternative considered: seed through infra bootstrap. That would put business
data in the infrastructure repository, violating the repository boundary.

## Risks / Trade-offs

- Stable UUID seed data can be tedious to maintain -> Keep the seed file small,
  explicit and reviewed with template references.
- Text enum persistence can drift from Java enums -> Validate enum values at
  application boundaries and keep OpenSpec as the source for allowed values.
- Spring Security may block API smoke tests by default -> Tests should either
  use Spring Security test support or document temporary test configuration
  without weakening production defaults.
- No local PostgreSQL is guaranteed in this repo -> Repository tests may need a
  documented test database strategy before implementation, or start with
  slice/domain/API tests that do not pretend persistence is validated.

## Migration Plan

1. Add the `exercise` module code.
2. Add schema and seed data in the product repository.
3. Add tests for domain validation and API behavior.
4. Add repository/integration tests only when a PostgreSQL test strategy is
   documented.
5. Run `openspec validate --specs`, `openspec validate implement-exercise-library`
   and Maven tests before implementation is considered complete.

Rollback is straightforward before production data exists: remove the module,
schema and seed files from the branch.

## Open Questions

- Should repository tests use Testcontainers, a local PostgreSQL profile or an
  infra-provided database? This must be documented before claiming persistence
  behavior is fully tested.
- Should API endpoints be temporarily accessible during early MVP development,
  or protected immediately by authentication once auth is specified?
