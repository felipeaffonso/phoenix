# Exercise Library

The exercise library contains the canonical global catalog of movements and
athlete-owned private exercises available to workout templates.

Implementation requirements are governed by the OpenSpec spec at
`openspec/specs/exercise-library/spec.md`. This document remains the product
blueprint and detailed catalog reference.

## MVP Requirements

- Store one canonical global exercise record per movement.
- Allow athletes to create private exercises for movements not covered by the
  global catalog.
- Support English and Portuguese display names.
- Classify movement pattern and body area.
- Capture equipment type.
- Capture safety notes relevant to lumbar spine and shoulder constraints.
- Allow workout templates to reference exercises by stable identity, not by
  copied text.

## Exercise Fields

| Field | Purpose |
|---|---|
| `id` | Stable internal identity. |
| `name` | Canonical English exercise name. |
| `name_pt` | Portuguese display name. |
| `pattern` | Movement pattern, such as horizontal push or knee extension. |
| `body_area` | Primary trained area. |
| `equipment` | Machine, cable, dumbbell, bodyweight or cardio. |
| `default_safety_notes` | Execution constraints reused by templates. |
| `active` | Whether the exercise can be selected in new templates. |
| `scope` | `GLOBAL` for catalog exercises or `USER` for private athlete exercises. |
| `owner_user_id` | Owner UUID for private exercises; empty for global exercises. |

## Module Boundary

The backend module name is `exercise`.

The Java package is:

```text
dev.affonsera.phoenix.exercise
```

The module owns:

- exercise domain model
- exercise repository
- exercise database table
- exercise read API
- initial exercise seed data
- private exercise ownership and visibility rules

The module does not own workout-template-specific sets, reps, rest times,
execution notes or progression rules.

## Domain Model

### Exercise

| Field | Type | Required | Rules |
|---|---|---:|---|
| `id` | UUID | yes | Generated by the application when creating seed or user-defined records. |
| `scope` | ExerciseScope | yes | `GLOBAL` for catalog exercises or `USER` for private athlete exercises. |
| `ownerUserId` | UUID | conditional | Required for `USER` exercises and empty for `GLOBAL` exercises. |
| `name` | String | yes | Canonical name, unique ignoring case within the global catalog or within the owner's private exercises. |
| `namePt` | String | yes | Portuguese display name. |
| `movementPattern` | MovementPattern | yes | One of the documented enum values. |
| `bodyArea` | BodyArea | yes | One of the documented enum values. |
| `equipmentType` | EquipmentType | yes | One of the documented enum values. |
| `defaultSafetyNotes` | String | no | Plain text, reused as the default for templates. |
| `active` | boolean | yes | Defaults to `true`. Inactive records remain readable for history. |

### Identity

- Use UUIDs for exercise identity.
- UUIDs are stable across API responses, workout templates and workout
  history.
- Do not expose database sequence IDs for exercises.

### Naming

- `name` is the canonical lookup name.
- `namePt` is the Portuguese display name.
- Both names are required for seed data.
- Additional aliases are out of scope for the first implementation.
- Private exercises may omit `namePt`; when omitted, Phoenix stores `name` as
  the Portuguese display name.

### Safety Notes

- Safety notes are plain text in the first implementation.
- They should be short enough to show during template editing.
- Structured safety tags may be added later only if documented first.

## Database Schema

The PostgreSQL table name is `exercises`.

| Column | PostgreSQL Type | Null | Constraints |
|---|---|---:|---|
| `id` | `uuid` | no | Primary key. |
| `scope` | `varchar(20)` | no | `GLOBAL` or `USER`. |
| `owner_user_id` | `uuid` | yes | Required for private `USER` exercises and empty for global exercises. |
| `name` | `varchar(120)` | no | Canonical exercise name. |
| `name_pt` | `varchar(160)` | no | Portuguese display name. |
| `movement_pattern` | `varchar(60)` | no | Must match `MovementPattern`. |
| `body_area` | `varchar(60)` | no | Must match `BodyArea`. |
| `equipment_type` | `varchar(60)` | no | Must match `EquipmentType`. |
| `default_safety_notes` | `text` | yes | Plain text safety guidance. |
| `active` | `boolean` | no | Defaults to `true`. |
| `created_at` | `timestamptz` | no | Defaults to current timestamp. |
| `updated_at` | `timestamptz` | no | Defaults to current timestamp. |

Required indexes:

```sql
create unique index ux_exercises_global_name_lower on exercises (lower(name)) where scope = 'GLOBAL';
create unique index ux_exercises_owner_name_lower on exercises (owner_user_id, lower(name)) where scope = 'USER';
create index ix_exercises_scope on exercises (scope);
create index ix_exercises_owner_user_id on exercises (owner_user_id);
create index ix_exercises_active on exercises (active);
create index ix_exercises_movement_pattern on exercises (movement_pattern);
create index ix_exercises_body_area on exercises (body_area);
```

Initial implementation may use SQL migrations or Spring Data JDBC schema
initialization. The schema must live in this repository, not in
`phoenix-infra`.

## API Contract

The first global catalog implementation is read-only. Athlete-owned private
exercise write contracts are part of the MVP Exercise Library capability and
use owner-scoped permissions.

### List Exercises

```http
GET /api/exercises
```

Query parameters:

| Parameter | Required | Values | Behavior |
|---|---:|---|---|
| `active` | no | `true`, `false` | Defaults to `true`. |
| `movementPattern` | no | `MovementPattern` value | Filters by movement pattern. |
| `bodyArea` | no | `BodyArea` value | Filters by body area. |
| `equipmentType` | no | `EquipmentType` value | Filters by equipment. |

Response:

```json
[
  {
    "id": "7f2b5a5e-1c2d-4d0f-8c71-6c3fdc85b2ab",
    "scope": "GLOBAL",
    "ownerUserId": null,
    "name": "Chest Press (Machine)",
    "namePt": "Supino na maquina / Chest press",
    "movementPattern": "HORIZONTAL_PUSH",
    "bodyArea": "CHEST",
    "equipmentType": "MACHINE",
    "defaultSafetyNotes": "Back supported, scapulae set, shoulder stays on bench.",
    "active": true
  }
]
```

### Get Exercise

```http
GET /api/exercises/{id}
```

Responses:

| Status | Meaning |
|---:|---|
| `200` | Exercise found. |
| `404` | Exercise does not exist. |

### Private Exercise Creation

Authenticated athletes can create private exercises for their own use. Private
exercise creation should use a friendly form: name, movement pattern, body area
and equipment are required; Portuguese name and safety notes are optional.

Private exercises are visible to their owner alongside global exercises. They
are not visible to other athletes and cannot be promoted to the global catalog
in the MVP.

### Error Format

Use Spring Boot's default problem response for validation and not-found errors
unless a shared API error contract is documented later.

## Validation Rules

- `name` must be trimmed, non-empty and at most 120 characters.
- `namePt` must be trimmed, non-empty and at most 160 characters.
- `movementPattern`, `bodyArea` and `equipmentType` must match documented enum
  values.
- `defaultSafetyNotes` may be null, but if present must be trimmed and at most
  2000 characters.
- `active` must never be null.

## Seed Data

Seed data is part of the product repository.

Rules:

- Seed records must use stable UUIDs documented in code or migration files.
- Re-running seed logic must be idempotent.
- Seed updates must not break existing workout template references.
- Seed names should match the initial catalog below.

## Normalization Notes

- Repeated exercises across templates must reference the same exercise record.
- Template-specific sets, reps, rest and execution notes belong to workout
  templates, not to the exercise library.
- Cardio and mobility can exist in the library only as selectable template
  activities for the MVP workout flow.
